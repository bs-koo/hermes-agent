# IQSbz × Hermes 봇 연동 — 가능성 분석 & 설계

> 작성일: 2026-06-17 | 상태: **기술적 가능성 확인 완료, 조직 협조(인증/엔드포인트)만 남음** | 대상: IQSbz4u(사내 근태·스케줄·손익 포털)

---

## 1. 결론 (TL;DR)

- **기술적으로 100% 가능.** 이미 구축된 Hermes 모니터링 봇에 **IQSbz 전용 도구(MCP)를 추가**하는 방식.
- 봇 자체(Gemini + Google Chat 게이트웨이)는 그대로 재사용 → "오늘 누가 휴가?" 같은 질문에 실시간 답변.
- **유일한 관건은 인증(JWT) 자동화 + 봇 전용 계정/엔드포인트 확보** = 코드보다 *조직 협조*가 핵심.

## 2. 확인된 사실 (브라우저 DevTools 조사 결과, 2026-06-17)

| 항목 | 값 |
|------|-----|
| 프론트엔드 | `https://iqsbz.sqisoft.com:8789` (SPA) |
| **API 서버** | **`https://iqsbz-api.sqisoft.com:9010`** (프론트와 분리된 REST API) |
| 호스팅 | AWS 서울 `52.78.185.141` → **공개 도메인, 네트워크 접근 제약 없음** |
| 데이터 형식 | **JSON** (깔끔. 예: `{"workTypeVo":{"workTime":"08002200","coreTime":"10001600",...}}`) |
| **인증 방식** | **JWT Bearer 토큰** — `Authorization: Bearer eyJhbG...` (HS256) |
| CORS | `Access-Control-Allow-Credentials: true`, Origin = 프론트 |
| 예시 엔드포인트 | `GET /it/.../model/250041` (개별 레코드 조회 패턴) |

→ **REST + JSON + 공개망 + 토큰 인증** = 봇 연동에 이상적인 조건.

## 3. 아키텍처 — 어떻게 작동하나

```
[Google Chat]  "오늘 우리 부서 누가 휴가야?"
      │
      ▼
[Hermes 봇 / Gemini]                    ← 이미 구축 완료 (재사용)
      │  (도구 호출)
      ▼
[IQSbz MCP 서버]                        ← 새로 만들 것
      │  1) 로그인: 서비스계정 ID/PW → JWT 발급
      │  2) JWT로 API 호출 (스케줄/휴가 조회)
      │  3) 토큰 만료(401) 시 자동 재로그인
      ▼
[iqsbz-api.sqisoft.com:9010]            ← IQSbz 백엔드
      │  JSON 응답
      ▼
[봇이 한국어로 요약 → Google Chat 답변]
```

→ **CloudWatch를 MCP로 붙인 것과 똑같은 패턴.** 데이터 소스만 AWS → IQSbz로 바뀜.

---

## 4. 인증 흐름 (핵심 설계 포인트)

- JWT는 **만료**된다 → 토큰을 코드/설정에 박아넣으면 곧 멈춤. **하드코딩 불가.**
- 따라서 MCP가 **스스로 로그인해 토큰을 받고, 만료되면 다시 받는** 구조여야 함:
  1. 서비스계정 ID/PW로 **로그인 API 호출** → JWT 수신
  2. JWT를 메모리에 캐시 → 모든 API 호출에 `Authorization: Bearer` 첨부
  3. 401(만료) 응답 시 → 재로그인 후 재시도
- (개발팀이 **장수명 토큰 / API 키**를 줄 수 있으면 1~3 단계가 더 단순해짐)

## 5. 만들 것 — IQSbz MCP 서버 (도구 예시)

| 도구 | 기능 | 사용 예 |
|------|------|---------|
| `get_today_leave(부서)` | 오늘 휴가/반차자 | "오늘 GX본부 누가 휴가?" |
| `get_dept_schedule(부서, 기간)` | 부서 근무 스케줄 | "이번주 사업개발팀 스케줄 요약" |
| `get_work_hours(직원, 월)` | 근무시간 현황 | "이번달 내 근무시간 얼마?" |
| `get_overtime(직원, 월)` | 초과근무 | "이번달 우리팀 초과근무" |

각 도구 = 해당 엔드포인트 호출 + JSON 정리 후 반환. 봇(Gemini)이 결과를 한국어로 요약.

## 6. 조직에서 받아야 할 것 (실질 블로커)

1. **봇 전용 계정** (ID/PW) — 개인 계정 비번을 봇에 넣는 건 보안상 비권장. *읽기 전용 권한*의 전용 계정 권장.
   - 또는 **봇용 API 토큰/키** (있으면 더 깔끔)
2. **로그인 API 스펙** — 어느 URL에 ID/PW를 어떻게 POST하면 JWT를 주는지 (요청/응답 형식)
3. **조회 엔드포인트 목록 + 응답 스키마** — 부서 스케줄/휴가/근태 데이터를 주는 API URL들
4. (선택) **API 문서 / Swagger** 있으면 1~3을 한 번에 해결

> 자사 제품이므로 "사내 모니터링 봇이 **근태/스케줄을 읽기 전용**으로 조회" 용도로 개발팀에 요청하면 받기 수월.

## 8. 고려사항 / 리스크

- **보안**: 봇 계정 권한 최소화(읽기 전용·필요 부서 한정). JWT/자격증명 안전 보관(.env, 평문 노출 금지).
- **권한 범위 = 응답 범위**: 봇 계정이 볼 수 있는 데이터만 봇이 답할 수 있음. (부서/전사 범위 정책 협의 필요)
- **운영 위치**: 봇 도는 곳(현재 로컬 PC / 향후 서버)에서 `iqsbz-api.sqisoft.com:9010` 접근 — 공개망이라 OK.
- **토큰 만료/갱신**: 견고하게 처리(401 → 재로그인). 로그인 실패/계정 잠김 대비.
- **개인정보**: 근태/휴가는 직원 개인정보 → 봇 노출 범위·로깅에 주의(사내 정책 준수).

## 9. 빠른 시연(PoC) 경로 — 원할 경우

- **현재 개인 JWT 토큰 + 스케줄 조회 엔드포인트 1개**만 있으면, "봇이 IQSbz 데이터를 실제로 읽어 답한다"를 **즉석 1회 시연** 가능.
- ⚠️ 단 그 토큰은 곧 만료 → "되는구나" 확인용. **실운영은 6번(서비스계정+로그인 자동화) 필요.**

## 10-B. 로그인 메커니즘 — 역공학 완료 (2026-06-23)

> 6번의 "로그인 API 스펙 미확보" 블로커 해소. 라이브 로그인 페이지(`iqsbz.sqisoft.com`) 직접 분석 결과.

- **로그인 페이지**: `https://iqsbz.sqisoft.com/` (ID + Password, 서버 Apache/스프링)
- **① 공개키**: `POST /getRsaPublicKey` → `{"publicKey": "<PEM>"}` (RSA-2048)
- **② 암호화**: 클라가 `jsencrypt@3.3.2`(RSA **PKCS#1 v1.5**)로 `email`·`password`를 각각 암호화 → base64.
  - ⚠️ **클라는 입력값을 그대로 암호화**한다(`email = $("#email").val()`). `@sqisoft.com` **자동 부착 없음** → 서버가 붙이거나 사용자가 전체 이메일 입력.
- **③ 로그인**: `POST /doLogin` (application/x-www-form-urlencoded)
  - body: `email=<RSA(email)>&password=<RSA(password)>`
  - 성공 → `302 /main` + `JSESSIONID` 쿠키 / 실패 → `302 /common/error401`
- **Python 복제 검증됨**: `cryptography`(load_pem_public_key + PKCS1v15)로 동일 요청 생성 시 서버 수용(더미 자격증명은 정상적으로 error401). → `iqsbz_auth_probe.py` 참조.
- **남은 미확인**: 로그인 성공 후 **:9010 API용 JWT 발급 시점**(세션 쿠키만으로 :9010 호출되는지 / 별도 토큰 발급 엔드포인트가 있는지) — 실제 자격증명 1회로 캡처 필요.
