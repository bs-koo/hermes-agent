# 손님이 추가로 해줘야 하는 설정 — 상세 가이드

> 대상: dataviz-prod (**Java/Tomcat on Elastic Beanstalk**, nginx 프록시 / RDS PostgreSQL `gseed-db` / 계정 062364668115 / ap-northeast-2)
> 각 설정 = "어떤 인사이트가 풀리나"와 짝지어 정리. 읽기 권한 밖이라 **손님/개발자만** 할 수 있는 것들.

## 우선순위 한눈에
| | 설정 | 누가 | 난이도 | 풀리는 기능 | 비용 |
|--|------|------|--------|-------------|------|
| **B** | 앱(Tomcat) 로그 스트리밍 | 개발자(재배포) | ★★ | 앱 예외/스택트레이스 분석 | $0~소액(로그저장) |
| **C** | RDS 느린쿼리 로깅 | 손님/DBA(콘솔) | ★★ | 느린 쿼리 Top 분석 | $0 |
| **D** | CloudWatch Agent | 개발자(재배포) | ★★★ | 메모리·디스크 사용률 | 소액(custom metric) |

> **A. 비용 인사이트는 제거됨** — Cost Explorer API가 호출당 $0.01 과금이라 비용 민감 방침상 미사용. (B·C·D만 진행)

## B. 앱(Tomcat) 로그 → CloudWatch 스트리밍 (개발자 재배포) ⭐가장 중요한 갭

**왜**: 지금 CloudWatch엔 nginx·eb·DB 로그만 있고 **앱(서버 프로세스)의 예외·스택트레이스가 없다.** Tomcat 앱 로그(`/var/log/tomcat/catalina.out`)를 올리면 "어떤 예외가 터졌나/신규 에러" 분석이 가능해진다.

**단계** (앱 소스 루트에 `.ebextensions` 파일 1개 추가 후 배포):

1. 앱 로그 경로 확인 — Tomcat 기본은 **`/var/log/tomcat/catalina.out`** (Spring Boot가 별도 logback 파일로 쓰면 그 경로로 변경).

2. 앱 소스에 **`.ebextensions/cloudwatch-app-logs.config`** 생성:
```yaml
# Amazon Linux 2 (Tomcat) — awslogs로 catalina.out 스트리밍
files:
  "/etc/awslogs/config/catalina.conf":
    mode: "000644"
    owner: root
    group: root
    content: |
      [/var/log/tomcat/catalina.out]
      log_group_name = /aws/elasticbeanstalk/dataviz-prod/var/log/tomcat/catalina.out
      log_stream_name = {instance_id}
      file = /var/log/tomcat/catalina.out
      datetime_format = %d-%b-%Y %H:%M:%S
container_commands:
  01_restart_awslogs:
    command: "systemctl restart awslogsd.service || service awslogs restart || true"
```
> ⚠️ **플랫폼 확인**: EB 콘솔 → 구성 → 플랫폼.
> - **Amazon Linux 2** → 위 awslogs 방식 그대로.
> - **Amazon Linux 2023** → awslogs 대신 CloudWatch Agent 사용(설정 형식 다름). 이 경우 알려주면 AL2023용 스니펫으로 맞춰드림.

3. `eb deploy` (또는 콘솔에서 새 버전 업로드).

4. 배포 후 `/aws/elasticbeanstalk/dataviz-prod/var/log/tomcat/catalina.out` 로그그룹이 생기면 성공 → 봇/리포트가 앱 에러를 분석.

**전제**: EB "인스턴스 로그 스트리밍(CloudWatch Logs)"이 켜져 있어야 함(이미 켜져 있음 — nginx 로그가 올라오는 중).

## C. RDS 느린 쿼리 로깅 (손님/DBA 콘솔)

**왜**: "DB를 무겁게 하는 느린 쿼리 Top N"을 분석하려면, PostgreSQL이 느린 쿼리를 로그에 남겨야 한다. postgres 로그는 이미 CloudWatch로 스트리밍 중(`/aws/rds/instance/gseed-db/postgresql`)이라, 파라미터만 켜면 바로 분석 가능.

**단계**:
1. RDS 콘솔 → 데이터베이스 → **gseed-db** → 구성 → **파라미터 그룹** 확인.
2. **default.postgresXX 면** (기본은 수정 불가):
   - 파라미터 그룹 → **생성** → 동일 family(예: postgres15) → 이름 `gseed-db-custom`
   - 만든 그룹 편집 → **`log_min_duration_statement` = `1000`** (1초 이상 쿼리 로깅; 500으로 더 민감하게도 가능)
   - gseed-db → 수정 → 파라미터 그룹을 `gseed-db-custom` 으로 변경 → 적용 → **재부팅 1회**(그룹 연결 반영)
3. **이미 커스텀 그룹이면**: 그 그룹에서 `log_min_duration_statement = 1000` 만 변경 (동적 적용, **재부팅 불필요**).
4. 끝. 이후 느린 쿼리가 postgres 로그에 남고 → 봇/리포트가 Top 분석.

> 값 절충: 1000ms = 1초↑ 쿼리만(로그 적음). 너무 낮추면(예: 100) 로그량·비용↑.

## D. 메모리·디스크 사용률 — CloudWatch Agent (개발자 재배포, 선택)

**왜**: EC2 기본 메트릭엔 **메모리%·디스크%가 없다**(CPU·네트워크·EBS IO만). 서버 메모리/디스크 포화를 보려면 CloudWatch Agent가 필요.

**단계**: 앱 소스에 **`.ebextensions/cloudwatch-agent.config`** 추가:
```yaml
# CloudWatch Agent 설치 + 메모리/디스크 사용률 발행 (AL2)
packages:
  yum:
    amazon-cloudwatch-agent: []
files:
  "/opt/aws/amazon-cloudwatch-agent/etc/eb-cw-agent.json":
    mode: "000644"
    owner: root
    group: root
    content: |
      {
        "metrics": {
          "append_dimensions": { "InstanceId": "${aws:InstanceId}" },
          "metrics_collected": {
            "mem": { "measurement": [ "mem_used_percent" ] },
            "disk": { "measurement": [ "disk_used_percent" ], "resources": [ "/" ] }
          }
        }
      }
container_commands:
  01_start_cw_agent:
    command: "/opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl -a fetch-config -m ec2 -c file:/opt/aws/amazon-cloudwatch-agent/etc/eb-cw-agent.json -s"
```
배포 후 `CWAgent` 네임스페이스에 `mem_used_percent`·`disk_used_percent` 발행 → 봇이 조회.
> 비용: custom metric당 ~$0.30/월. 2개면 무시 가능 수준.

## (선택) 그 외 — 필요해지면

| 원하는 것 | 필요 설정 | 비고 |
|-----------|-----------|------|
| RDS 용량 **%** (여유공간 비율) | IAM에 `rds:DescribeDBInstances` 추가 | 총 할당량(분모) 확보용 |
| 인스턴스 목록/디스커버리 | IAM에 `ec2:DescribeInstances`, `elasticbeanstalk:Describe*` | 봇이 리소스 자동 파악 |
| 외부 24/7 가동률 감시 | Synthetics 캐너리(`synthetics-setup-가이드.md`) | 현재는 $0 PC 핑으로 대체 중 |
| per-URL 응답시간(전수) | nginx 로그포맷에 `$request_time` + 재배포 | 앱 수정이라 보류 |
| AWS 밖 다른 서버 로그 | 그 서버에 CloudWatch Agent + IAM | 서버 접근 필요 |

## 추천 순서
1. **C(느린쿼리)** — 손님이 콘솔에서, DB 성능 분석 강화.
2. **B(앱 로그)** — 개발자에게 `.ebextensions` 요청 → 앱 에러/스택트레이스 분석 확보(가장 가치 큼).
3. **D(메모리/디스크)** — 여유 될 때.

> (비용 인사이트 A는 Cost Explorer 과금 때문에 제거 — 진행 안 함)
