# dataviz-prod 운영 대시보드 — 구축 기록 (전체)

> 작성일: 2026-06-22 | 대상: dataviz-prod (AWS ap-northeast-2) | 담당: bonseung@sqisoft.com
> 기존 "Hermes 운영 모니터링 봇"(Google Chat) 위에, 운영 상태를 **브라우저 웹 대시보드 + 운영봇 채팅**으로 보는 시스템을 추가 구축한 전체 기록.
> 작업 방식: `/oh-my-gx:gx-dev` 파이프라인(PRD→설계→구현→리뷰→완료) + 이후 사용자 피드백 반영 다수 반복.

---

## 0. 한 줄 요약

기존 봇이 Google Chat 텍스트로만 주던 운영 정보를, **단일 Docker 컨테이너(FastAPI + 백그라운드 수집기 + SQLite 캐시)** 로 패키징한 **멀티뷰 웹 대시보드**로 확장. AWS 무료 메트릭(알람·가동률·트래픽·품질·EC2 6대·RDS 3대·CloudFront 2배포)을 시각화하고, **Gemini 운영봇 채팅**을 우측에 붙임. 비용 $0 유지. 로컬 Docker 실행, 회사 서버 이관 시 동일 이미지 재사용.

## 1. 작업 타임라인 (이 세션)

1. **git 편입 + 시크릿 분리**: 기존 hermes_agent를 git 레포로 초기화(remote `bs-koo/hermes-agent`). webhook 시크릿을 `.env`(`GCHAT_WEBHOOK`)로 분리하고 4개 스크립트에서 하드코딩 제거(`_env.py` 공용 로더). 초기 커밋 `4fae921`.
2. **gx-dev 파이프라인**: PRD 확정 → 설계(대형, design-critic 검토) → 구현(B1 백엔드 / B2 프론트·Docker) → 리뷰(qa-manager + security-auditor) → 인수검증 ACCEPT → 커밋 `4607071`.
3. **확장·개선 반복**(피드백 기반): 채팅 패널 추가 → 디자인 4회 변경(Apple→AWS→Friday→토스) → 멀티뷰 라우팅 → 차트 툴팁/가독성 → 1단계(호스트 EC2 + DB 심화) → 2단계(CloudFront CDN) → EC2/RDS 메타·전체 인스턴스 → 알람 상세·용어 설명·응답품질 에러중심 등.

## 2. 아키텍처

```
┌─────────────── Docker 컨테이너 (포트 8080, --workers 1) ───────────────┐
│ uvicorn dashboard.api:app                                              │
│  ┌── FastAPI(메인 스레드) ──┐     ┌── Scheduler(데몬 스레드 1개) ──┐    │
│  │ /api/* (SQLite만 읽음)    │     │ 60s tick, 잡별 try/except 격리 │    │
│  │ /api/chat (Gemini)        │◀─┐  │ alarms5m uptime5m traffic10m   │    │
│  │ / (정적 프론트)            │  │  │ db10m host10m cdn10m + purge   │    │
│  └──────────┬───────────────┘  │  └────────────┬───────────────────┘    │
│             │ SELECT            │               │ upsert/append          │
│        ┌────▼───────────────────▼───────────────▼──┐                    │
│        │ SQLite (/db/dashboard.db, WAL) ← named vol │                    │
│        └────┬───────────────────────────────────────┘                   │
└─────────────┼────────────────────────────────────────────────────────────┘
   ┌──────────▼─────────┐  ┌──────────────┐  ┌────────────────────┐  ┌──────────┐
   │ CloudWatch         │  │ Logs Insights│  │ /data/uptime_log.csv│  │ Gemini   │
   │ DescribeAlarms     │  │ (nginx 로그) │  │ (가동률 핑, ro)     │  │ 2.5 Flash│
   │ GetMetricData      │  └──────────────┘  └────────────────────┘  │(채팅)    │
   │ EC2/RDS Describe*  │                                            └──────────┘
   └────────────────────┘
```

- **읽기/수집 분리**: 백그라운드 수집기가 주기마다 AWS/CSV → SQLite 적재. FastAPI는 **AWS 미호출, SQLite만 읽어** 즉시 응답(화면 로딩 3초 이내).
- **단일 워커 강제**(`--workers 1`): 멀티워커 시 스케줄러가 워커마다 떠서 Logs Insights 중복 발사·SQLite 락 → 금지.
- **수집 주기**: 알람 5분 / 가동률 5분 / 트래픽 10분 / DB 10분 / 호스트 10분 / CDN 10분. SQLite 30일 보관(시계열만 purge, 스냅샷은 교체).
- **이관 일관성**: 로컬은 `~/.aws` 마운트(`hermes-cw` 프로파일), 회사 EC2 이관 시 마운트 빼고 IAM 역할. CSV/DB 경로는 env(`UPTIME_CSV`/`DASH_DB`)로 흡수.

## 3. 데이터 소스 (전부 무료 AWS / $0)

| 영역 | 소스 | 내용 |
|------|------|------|
| 알람 | CloudWatch DescribeAlarms | 4개 알람 상태 + **설명·임계치·조건·메트릭·현재이유** |
| 가동률·응답시간 | uptime_log.csv(호스트 핑) | health/home 가동률%, 응답시간 avg/p95 (1h 버킷) |
| 트래픽·사용자 | nginx access.log (Logs Insights) | 실사용자 요청수·사용자(XFF)·Top API·시간대 추세 (핑/스캐너 제외) |
| 응답품질 | nginx access.log | 상태코드 분포·**에러 URL(어디서)**·스캐너 탐침 분리 |
| 호스트(EC2) | CloudWatch AWS/EC2 + EC2 DescribeInstances | **6개 인스턴스** CPU·네트워크·EBS IO·CPU크레딧·상태 + **IP·이름·타입** |
| DB(RDS) | CloudWatch AWS/RDS | **3개 인스턴스** CPU·메모리·연결·지연·DBLoad·IOPS·여유공간 + 시계열 |
| CDN | CloudWatch AWS/CloudFront(us-east-1) | **2개 배포** 요청수·전송량·4xx/5xx/전체 에러율 + 시계열 |
| 채팅 | Gemini 2.5 Flash(무료키) | SQLite 캐시 데이터를 컨텍스트로 한국어 Q&A |

**제약(현재 불가, 정직)**: per-endpoint 실응답시간(앱에 nginx `$request_time` 추가 필요), EC2 메모리%(CloudWatch Agent 재배포), WAS 내부(JVM 힙/GC/스레드 — JMX/Actuator 계측 필요), 비용(Cost Explorer 호출당 $0.01 과금 → 제외). postgres 느린쿼리는 RDS 파라미터(`log_min_duration_statement`) 활성화 필요.

## 4. 화면 구성 (멀티뷰 hash 라우팅)

좌측 사이드바 네비 + 중앙 메인 뷰 + 우측 운영봇 채팅(3컬럼). 각 메뉴 클릭 시 해당 상세 뷰로 전환.

| 라우트 | 뷰 | 내용 |
|--------|-----|------|
| `#/dashboard` | 개요 | KPI 요약 5칸 + 영역별 미니카드(클릭 시 상세 진입) |
| `#/alarms` | 알람 | 알람별 카드(이름·상태·설명·조건·현재이유) |
| `#/uptime` | 가동률·응답시간 | 가동률% 요약(sparkline) + 응답시간 메인 차트(avg/p95) |
| `#/traffic` | 트래픽·사용자 | 실사용자 요청·사용자·Top API 막대·시간대 추세 |
| `#/quality` | 응답품질 | 에러 URL 중심(어디서·상태·건수) + 스캐너 분리 |
| `#/host` | EC2 인스턴스 | 6개 카드(이름/IP/타입/CPU/네트워크/크레딧) + 클릭 시 상세 모달(시계열) |
| `#/cdn` | CloudFront CDN | 배포별 요청·전송량·에러율 + 시계열 |
| `#/database` | DB 성능 | RDS 인스턴스별 카드 + 시계열(CPU/메모리/DBLoad/연결) |

**상단 헤더**: 서비스명 + ● Operational 상태배지(알람 경보 시 빨강) + refresh + 마지막 갱신.

## 5. 운영봇 채팅 (Gemini)

- 우측 사이드바 고정. 사용자 질문 → `POST /api/chat` → SQLite 캐시(알람/가동률/트래픽/DB 요약)를 컨텍스트로 Gemini 2.5 Flash가 한국어 답변.
- urllib REST 직접 호출(의존성 무추가), 일시적 5xx/429는 점증 백오프 재시도.
- 예시 질문 칩("지금 알람 상태", "이번주 많이 쓴 API", "DB 상태 어때?") 클릭 자동 전송. typing 인디케이터.
- 키: 기존 hermes의 `GEMINI_API_KEY`(Google AI Studio 무료) 재사용.

## 11. 트러블슈팅 (구축 중 해결한 핵심 이슈)

| 문제 | 원인 | 해결 |
|------|------|------|
| period=7 호버 422 | FastAPI `Literal[1,7,30]`가 쿼리문자열 "7" coerce 실패 | `int` + `_clamp_period()` 화이트리스트 |
| 차트 호버 툴팁 안 뜸 | Chart.js v4는 `Chart.defaults.interaction={}` 직접할당 무효 | `Chart.defaults.set('interaction'/'plugins.tooltip', …)` |
| 변경이 화면에 반영 안 됨 | 브라우저(playwright) 캐시가 옛 JS/CSS 사용 | 정적 자산 `Cache-Control: no-store`(chart 번들 제외) |
| 비root 컨테이너 SQLite write 실패 | named volume이 이전 root 소유 | 빈 볼륨 재생성(이미지 /db app 소유 상속) |
| 비root에서 hermes-cw 프로파일 못 찾음 | `/root/.aws` 경로가 app 사용자와 불일치 | `AWS_CONFIG_FILE`/`AWS_SHARED_CREDENTIALS_FILE` env로 명시(`/aws` 마운트) |
| 기동 직후 첫 수집 빈 데이터 | list_metrics/describe 일시 빈/타이밍 | 스케줄러 기동 5초 지연 + **빈 결과 시 캐시 보존(저장 skip)** |
| EC2/RDS 엉뚱한 인스턴스 | list_metrics 첫 값만 사용 | 전체 인스턴스 수집 + config 우선 폴백 |
| 트래픽 14건 "버그?" | 전체 288건 중 274가 가동률 핑(UA 제외) | 버그 아님 — 실사용자 14건. "실사용자 요청" 표기로 명확화 |
| Gemini 채팅 503 | Gemini 서버 일시 과부하 | 5xx/429 점증 백오프 재시도 |
