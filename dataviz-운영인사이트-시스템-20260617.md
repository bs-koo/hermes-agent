# dataviz-prod 운영 인사이트 시스템 — 구축 문서

> 작성일: 2026-06-17 | 대상: dataviz-prod (Elastic Beanstalk + CloudFront + RDS, ap-northeast-2)
> 기존 "알람 모니터링 봇" 위에 **운영 인사이트(트래픽·사용자·품질·비용)** 를 얹은 확장 작업 기록.

## 1. 한눈에 — 무엇이 추가됐나

기존 봇은 **알람 발생/정상 여부**만 봤다. 이번 확장으로 다음 인사이트가 추가됐다:

| # | 인사이트 | 출처 | 상태 |
|---|----------|------|------|
| ② | **사용자·트래픽** (인기 화면, 활성 사용자, 트래픽 추세) | nginx access.log | ✅ 작동 |
| ④ | **에러·품질** (4xx/5xx URL, 상태분포, 스캐너 노이즈) | nginx access.log | ✅ 작동 |
| ③ | **성능·부하** (DB 지연·부하·연결·서버 CPU) | RDS/EC2 메트릭 | ✅ 부분 (AWS-side 간접 측정) |
| ~~①~~ | ~~비용~~ | — | ❌ **제외** (Cost Explorer API 호출당 $0.01 과금 → 미사용, 5장) |

> ③ 성능: **앱을 건드리지 않는다**는 제약 하에, per-URL 실사용자 응답시간(passive)은 측정 데이터가 없어 불가(4.5장). 대신 **RDS 성능 메트릭(지연·부하·연결)+EC2 부하**로 "느린가/병목이 어디인가"를 AWS-side만으로 측정한다. 진짜 per-endpoint 응답시간이 필요하면 Synthetics 캐너리(능동 측정, 8장).

전달 방식 2가지:
- **온디맨드 Q&A** — Google Chat에서 봇에게 물으면 즉석 집계 답변 (②④)
- **주간 자동 리포트** — 매주 월요일 09:00 스페이스로 자동 전송 (②④ + ① 권한 시)

## 2. 아키텍처 (데이터 흐름)

```
[Google Chat] ──질문──► [Hermes 봇/Gemini] ──► [CloudWatch MCP]
   "이번주 많이 쓴 화면?"        │                  │ execute_log_insights_query
                                 ▼                  ▼
                          한국어 요약 답변   /aws/.../nginx/access.log (Logs Insights 집계)

[매주 월 09:00] Windows 예약작업 ─► docker(cloudwatch-mcp) ─► weekly_insight.py
                                                                   │ boto3: Logs Insights + (Cost Explorer)
                                                                   ▼
                                                          Google Chat 스페이스(webhook)
```

- **로그가 금광**: `dataviz-prod` nginx access.log가 CloudWatch에 쌓여 있고(177MB), 현재 읽기전용 권한으로 바로 조회된다. 한 줄에 **URL·상태·바이트·referer·User-Agent·실제IP(XFF)** 가 모두 들어있다.
- **CloudFront 액세스 로그/권한 없이도** 트래픽·에러 인사이트가 전부 나온다(위 nginx 로그로 충분).

### nginx access.log 형식
```
3.172.65.19 - - [16/Jun/2026:02:41:54 +0000] "GET /api/charts?page=0&size=4 HTTP/1.1" 200 204 "https://app.gx-viz.com/home" "Mozilla/5.0 ..." "121.133.249.165"
```
파싱(Logs Insights):
```
parse @message '* - - [*] "* * *" * * "*" "*" "*"' as ip, ts, method, url, proto, status, bytes, referer, ua, xff
```
⚠️ **응답시간($request_time)은 이 포맷에 없음** → per-URL 지연 인사이트는 nginx 로그포맷 1줄 추가가 선행돼야 함(8장 참고).

## 3. 실측 진단 결과 (2026-06-17, 최근 8일)

라이브로 돌려 확인한 실제 데이터:

- **앱 정체**: `app.gx-viz.com` (GX 데이터 시각화 앱, Spring 백엔드, Google OAuth 로그인)
- **트래픽**: 총 31건/8일, 전주(95건) 대비 -67% → **내부용 저트래픽 앱**
- **인기 API**: `/api/gx-templates`, `/api/chart-templates`, `/api/charts`, `/api/auth/refresh`
- **활성 사용자**: 실IP 5개, 사실상 내부 2명(`121.133.249.165`, `182.172.43.169`)이 주 사용
- **품질**: 2xx 21 / 3xx 4 / 4xx 6 / 5xx 0 (5xx 없음 = 서버 에러 양호)
- **스캐너 탐침 발견**: `/api/.git/config`, `/actuator/env` → 외부 자동 스캐너의 정보노출 시도(앱이 401/404로 차단). "봇/스캐너 노이즈"로 자동 분류됨.

### 발견된 인프라 사실
- RDS: **`gseed-db`** (PostgreSQL), Enhanced Monitoring(RDSOSMetrics) 켜짐, postgres 로그 실시간 수집(103MB)
- EB 환경 2개: `dataviz-prod`, `sqi-comn-term-env`
- **Synthetics 캐너리**(`cwsyn-peis_naver`) 가동 중 → 가동률(uptime) 데이터 존재
- ALB 없음(단일 인스턴스 EB) → CloudFront가 EB로 직접 연결
- 기존 알람 4개: cloudfront-5xx(데이터부족), eb-health-severe(OK), rds-high-cpu(OK), rds-low-storage(OK)

## 4.5 ③ 성능 — 서버 무수정·AWS-side로 어디까지 (2026-06-17 진단)

**제약: 운영 앱(nginx) 무수정.** 이 조건에서 AWS만으로 가능한 성능 측정의 실체를 진단함:

| 후보(AWS-side) | 결과 | 이유 |
|----------------|------|------|
| Synthetics 캐너리 지연 | ❌ 메트릭 없음 | 계정 내 캐너리(`cwsyn-peis_naver`)는 dataviz 대상 아님 |
| EB 지연 퍼센타일(ApplicationLatencyP*) | ❌ 미발행 | EB enhanced health CW 메트릭 미활성(EnvironmentHealth만) |
| X-Ray per-endpoint 지연 | ❌ 트레이스 없음 | 앱 미계측 |
| ALB TargetResponseTime | ❌ 없음 | 단일 인스턴스 EB(ALB 없음) |
| CloudFront OriginLatency | ❌ 없음 | additional metrics 미활성 |
| **RDS 성능 메트릭** | ✅ **풍부** | CPU·**DBLoad(PI 켜짐)**·연결·ReadLatency/WriteLatency·IOPS·DiskQueue |
| **EC2/EBS 부하** | ✅ 있음 | CPUUtilization·Network·EBS IO |

**결론**: **passive(실사용자) per-URL 응답시간은 데이터 자체가 없음** — 앱 계측(X-Ray/nginx 응답시간) 또는 ALB 없이는 불가. 대신 **"느린가/병목이 어디인가"** 는 RDS 성능으로 상당 부분 답 가능(웹앱 병목은 대개 DB).

→ **채택**: 주간 리포트 + 봇 온디맨드에 **RDS 성능/부하 + EC2 부하** 섹션 추가. 실측값(최근 8일):
```
🗄️ DB 성능/부하 (gseed-db, AWS 메트릭)
  CPU 평균 3.8% · 피크 7.7%
  연결 평균 5 · 피크 9
  쿼리지연(평균) 읽기 0.14ms · 쓰기 0.56ms
  DB부하(활성세션) 평균 0.00 · 피크 3
  디스크큐 피크 0.16 · 여유공간 최소 49.3GB
```
→ 해석: DB는 매우 여유(CPU 4%, 지연 1ms 미만). 현재 성능 병목 징후 없음.

**진짜 per-endpoint 응답시간을 원하면(앱 무수정)**: **CloudWatch Synthetics 캐너리** = AWS가 주기적으로 dataviz API(`/api/charts` 등)를 능동 호출해 응답시간·가용성을 측정. 앱 소스는 안 건드림. 단 셋업 필요(synthetics 쓰기권한 + IAM 역할 + S3 버킷, 비용 ~$0.0012/run). 8장 참조.

## 6. 주간 자동 리포트 — weekly_insight.py

- **무엇**: nginx access.log를 Logs Insights로 집계 → 트래픽/사용자/품질/DB성능/가동률 한국어 리포트 → Google Chat 전송
- **언제**: 매주 **월요일 09:00** (Windows 예약작업 `Hermes-WeeklyInsight`)
- **파일**:
  - `weekly_insight.py` — 집계·메시지 생성·전송 본체
  - `run_weekly_insight.cmd` — docker 실행 래퍼 (POST=1, DAYS=7)
  - `weekly_hidden.vbs` — 콘솔 숨김 실행
- **환경변수**: `POST=1` 실제 전송 / (기본 0: 출력만, 테스트용), `DAYS=7` 집계 기간

### 리포트 샘플
```
📊 dataviz-prod 주간 인사이트 · 06/09~06/17 (최근 8일)
🌐 트래픽  총 31건 · 일평균 3건 · 전주 대비 -67%
👥 사용자  활성 클라이언트 5명 · 121.133.249.165(13), 182.172.43.169(13), ...
🔥 많이 쓴 화면/API
  • GET /api/gx-templates — 5
  • GET /api/chart-templates — 4 ...
✅ 응답 품질  2xx 21 / 3xx 4 / 4xx 6 / 5xx 0
  ⚠️ 에러 URL: 401 /api/auth/refresh (2) ...
  🤖 스캐너/봇 탐침 2건 무시(.git·actuator 등)
📈 가동률·응답시간  가동률 100% · health 평균 194ms · 홈 102ms

### 수동 실행/테스트
```
# 출력만(전송 안 함)
docker run --rm -v C:/Users/SQI/.aws:/root/.aws:ro -v D:/SQ/hermes_agent:/work:ro \
  -e AWS_PROFILE=hermes-cw -e AWS_REGION=ap-northeast-2 -e DAYS=7 \

## 8. 더 확장하려면 (보류/후속)

| 원하는 것 | 필요 작업 | 비고 |
|-----------|-----------|------|
| **per-endpoint 응답시간**(앱 무수정·AWS-native) | **Synthetics 캐너리** 생성(주요 API 능동 호출) | synthetics 쓰기권한+IAM역할+S3, ~$0.0012/run. 진짜 응답시간 측정 |
| per-URL 응답시간(passive, 전수) | EB nginx 로그포맷에 `$request_time` 추가 후 재배포 | 앱 수정 필요라 보류 선택됨 |
| 집계 지연 퍼센타일(P50/P90/P99) | EB enhanced health CW 메트릭 토글(콘솔, 소스 무수정) | 환경 설정 변경. 집계값만(per-URL 아님) |
| **온디맨드 비용 질문** | cost-explorer MCP 서버 추가 등록 | 주간 리포트로 우선 충당 |
| **RDS 용량 %** | `rds:DescribeDBInstances` 권한 + 총 할당량 분모 | FreeStorageSpace는 절대 바이트만 줌 |
| **느린 쿼리 Top SQL** | RDS Performance Insights 또는 log_min_duration_statement | postgres 로그/PI 활용 |
| **지역·디바이스 분포** | CloudFront 액세스 로그 활성화(S3) | nginx 로그엔 지역 정보 없음 |
| **가동률(uptime) %** | Synthetics 캐너리(cwsyn-peis_naver) 메트릭 집계 | 이미 데이터 존재 |

## 9. 운영 메모

- **봇/게이트웨이**: Startup 폴더 등록으로 로그인 시 자동 시작(영구화 완료). SOUL.md 변경은 게이트웨이 재시작 또는 세션 리셋(매일 04시/유휴 24h) 후 반영.
- **Docker 의존성(중요)**: CloudWatch 기능(봇의 알람/메트릭 답변, 10분 알람점검, 주간 리포트)은 `cloudwatch-mcp` 컨테이너에 의존 → **Docker가 꺼지면 이 기능들이 멈춤**(가동률 핑은 호스트 python이라 영향 없음). 2026-06-18 Docker Desktop을 **로그인 자동시작**으로 등록(`시작프로그램\Docker-Desktop.lnk`). 로그인 후 엔진 기동에 ~10~60초 소요.
- **저트래픽 주의**: dataviz는 내부용이라 건수가 작다. 리포트가 "0건/소량"이어도 정상 — 기간을 7일 이상으로 본다.
- **비용**: **전부 $0** (Logs Insights 스캔량 소량, 예약작업/웹훅/PC핑 무료). 비용 인사이트(Cost Explorer)는 호출당 $0.01 과금이라 **제거함**(5장).
- **관련 파일**: `insight_probe.py`(형식 샘플러), `insight_poc.py`(PoC 집계), `weekly_insight.py`(본체 — 트래픽·품질·DB성능·가동률), `uptime_ping.py`($0 가동률 핑), `canary-dataviz.js`·`synthetics-setup-가이드.md`(캐너리 옵션, 미사용).

### 가동률·응답시간 — $0 DIY PC 핑 (채택, 가동 중)
- **결정**: 비용 민감 + 어차피 PC 기반(봇과 동일 가동조건) → **AWS Synthetics 캐너리(월 ~$2~13) 대신 $0 DIY 핑** 채택.
- **구조**: `uptime_ping.py`(호스트 python, AWS 불필요)가 **5분마다** `/actuator/health`·홈을 HTTP GET → 응답시간·생존을 `uptime_log.csv`에 기록.
  - **다운/복구 '전환' 시에만** Google Chat 알림(스팸 방지, 상태는 `uptime_state.json`).
  - 스케줄: `Hermes-UptimePing`(5분), `run_uptime_ping.cmd` + `uptime_hidden.vbs`.
- **자동 표시**: 주간 리포트 `📈 가동률·응답시간` 섹션(가동률%·health 평균/P95·홈 평균). 실측 예: 가동률 100%, health 282ms, 홈 118ms.
