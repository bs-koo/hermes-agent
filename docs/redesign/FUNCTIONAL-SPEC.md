# dataviz-prod 운영 대시보드 — 기능 명세 (디자인 인수인계용)

> 이 문서는 **"무슨 기능이 있고 / 데이터가 어떻게 들어오고 / 무엇을 보여줘야 하는지"**만 담는다.
> **화면 레이아웃·메뉴 배치·차트 종류·채팅 UI·컴포넌트 형태는 일절 규정하지 않는다.**
> 정보구조(IA), 메뉴/화면 구성, 시각화 방식, 채팅 UI는 **전적으로 디자이너(Claude Design)에게 위임**한다.
> 시각 언어(색·타이포·간격·톤)는 별도 문서 `TOSS-DESIGN.md`를 따른다.

---

## 0. 한 줄 정의

AWS에 떠 있는 운영 서비스(`dataviz-prod`, ap-northeast-2)의 상태를 한 화면에서 보는 **읽기 전용 운영 모니터링 도구**.
운영자가 답해야 하는 단 하나의 질문 — **"지금 문제 있나? 있으면 무엇이고 얼마나 급한가?"** — 에 가장 빨리 답하는 것이 목적이다.

핵심 의도(우선순위):
1. **이상 신호 우선.** 정상일 땐 조용하고, 문제가 있을 때 그게 가장 먼저 눈에 들어와야 한다.
2. **숫자는 근거와 함께.** 모든 수치는 "언제 기준 / 무슨 임계 / 왜"가 따라붙는다.
3. **빠른 응답.** 화면은 항상 캐시만 읽어 3초 내 응답(아래 데이터 파이프라인 참조).

## 1. 데이터 파이프라인 (어떻게 들어오나)

```
외부 소스                         수집(백그라운드 데몬)          저장             화면
─────────────────────────────────────────────────────────────────────────────────────
AWS CloudWatch                ┐
  · 알람 상태(DescribeAlarms) │
  · EC2/RDS/CloudFront 메트릭  │   Scheduler 가 주기마다
    (GetMetricData)           ├──▶ 수집기 7종 실행 ──▶ SQLite 캐시 ──▶ REST API(JSON) ──▶ 프런트
  · 접근로그 분석             │      (각자 독립·예외격리)    (8 테이블)    (캐시만 읽음,    (폴링)
    (Logs Insights)           │                                          AWS 미호출)
나의 신서틱 핑 CSV            ┘
  (uptime_log.csv)
```

**중요한 설계 원칙**
- 화면/REST API는 **AWS를 직접 호출하지 않는다.** 오직 SQLite 캐시만 읽는다 → 항상 빠르고, AWS 장애와 무관하게 마지막 수집값을 보여줄 수 있다.
- 실제 수집은 **백그라운드 스케줄러**가 전담한다. 화면은 "마지막으로 성공한 수집"을 본다.
- 채팅 답변과 인사이트의 AI 요약만 요청 시점에 외부 LLM(Gemini)을 부른다. **이것도 근거는 캐시 데이터뿐.**

**수집 주기 / 보관**

| 도메인 | 출처 | 주기 | 데이터 성격 |
|---|---|---|---|
| 알람 | CloudWatch DescribeAlarms | 5분 | 상태 스냅샷 + 상태전이 이력 |
| 가동률·응답시간 | 신서틱 핑 로그(CSV) | 5분 | 시계열(버킷) |
| 트래픽·사용자 | CloudWatch Logs Insights(접근로그) | 10분 | 기간 스냅샷(1/7/30일) |
| 응답품질(상태코드/에러) | 위 트래픽 스냅샷에서 파생 | 10분 | 기간 스냅샷 |
| EC2 호스트 | CloudWatch GetMetricData | 10분 | 인스턴스별 스냅샷 + 시계열 |
| CDN(CloudFront) | CloudWatch GetMetricData(us-east-1) | 10분 | 배포별 스냅샷 + 시계열 |
| DB(RDS) | CloudWatch GetMetricData | 10분 | 인스턴스별 스냅샷 + 시계열 |

- 모든 시각은 **KST(UTC+9)** 로 표기.
- 시계열 보관 **30일**.
- 각 수집 잡은 **마지막 성공시각/상태**를 기록한다 → 수집이 밀리면(주기×2 초과) **"오래된 데이터(stale)"** 로 표시해야 한다(아래 §7).

## 2. 기능 도메인 (무엇을 감시하나)

각 도메인은 **"운영자가 답을 원하는 질문 / 보유한 데이터 필드·시계열 / 정상·주의·위험의 의미"** 세 가지로 기술한다.
**어떤 차트·카드·표로 그릴지는 디자이너가 정한다.**

### 2.1 알람 (CloudWatch Alarms)
- **답해야 할 질문:** 지금 울리고 있는 경보가 있나? 무슨 조건을 어겨서?
- **데이터(알람별):** 이름 · 상태(`ALARM`/`OK`/`INSUFFICIENT_DATA`) · 상태사유 · 메트릭 · 통계 · 기간 · 비교연산자(>, ≥, < …) · 임계값 · 조립된 조건문자열(예: `CPUUtilization > 80 (Average, 300s)`) · 상태변경시각 · 설명.
- **추가:** 알람별 **상태전이 이력**(상태, 전이시각의 시퀀스) — 시간에 따라 어떻게 켜지고 꺼졌는지.
- **상태 의미:** `ALARM`=위험 / `INSUFFICIENT_DATA`=메트릭 미수신(정보) / `OK`=정상.

### 2.2 가동률·응답시간 (신서틱 모니터링)
- **답해야 할 질문:** 서비스가 떠 있나(가동률)? 사용자 체감 속도는?
- **대상:** 엔드포인트별(예: `health`=헬스체크, `home`=홈페이지).
- **데이터(엔드포인트 × 시간버킷):** 성공률(%) · 평균 응답시간(ms) · p95 응답시간(ms, 가장 느린 5% 기준 체감지연).
- **요약:** 최근 24시간 성공률(엔드포인트별 성공/시도 합산).
- **기간 전환:** 1일 / 7일 / 30일.
- **상태 의미:** 성공률 100%에 가까울수록 양호 / 응답시간 급등은 주의.

### 2.3 트래픽·사용자
- **답해야 할 질문:** 얼마나 많이 쓰이나? 누가(몇 명) 어디를(어떤 API/화면) 많이 쓰나? 언제 몰리나?
- **데이터(기간 스냅샷):** 총 요청수 · 순 사용자수 · **상위 엔드포인트 랭킹**(메서드+URL, 히트수) · **시간대별 요청 추세** · 스캐너성 요청수 · 헬스체크 요청수(노이즈 분리용).
- **기간 전환:** 1일 / 7일 / 30일.

### 2.4 응답품질 (상태코드 / 에러)
- **답해야 할 질문:** 에러가 나고 있나? 어디서(어떤 URL) 무슨 코드로?
- **데이터(기간 스냅샷):** **상태코드 분포**(2xx/3xx/4xx/5xx/기타 집계) · **에러 발생 위치 랭킹**(상태코드+URL+건수, 4xx/5xx 중심).
- **상태 의미:** 5xx=서버에러(위험) / 4xx=클라이언트에러(주의) / 2xx=정상.

### 2.5 EC2 호스트
- **답해야 할 질문:** 서버 자원이 버티나? CPU/네트워크/디스크/크레딧 여유는?
- **대상:** 계정 전체 인스턴스(자동 발견), 주(primary) 인스턴스 표식 있음.
- **데이터(인스턴스별 스냅샷):** CPU 평균/최대(%) · 네트워크 in/out · EBS read/write · CPU 크레딧 잔량(t계열) · 상태검사 실패 여부 · 메타(사설IP·이름·타입·상태).
- **시계열:** CPU · 네트워크 in · 네트워크 out(상세 조회용).
- **상태 의미:** CPU 지속 고부하·상태검사 실패=위험 / 순간 피크·크레딧 소진 임박=주의.

### 2.6 CDN (CloudFront)
- **답해야 할 질문:** CDN이 정상 응답하나? 에러율·트래픽은?
- **대상:** 계정 전체 배포(자동 발견).
- **데이터(배포별 스냅샷):** 요청수 · 다운/업로드 바이트 · 4xx/5xx/전체 에러율(%).
- **시계열:** 요청수 · 전체 에러율.
- **상태 의미:** 5xx 에러율 상승=위험 / 4xx 에러율 상승=주의.

### 2.7 DB 성능 (RDS)
- **답해야 할 질문:** DB가 병목인가? CPU/연결/저장공간/부하는?
- **대상:** 계정 전체 RDS 인스턴스, 주(primary) DB 표식 있음.
- **데이터(인스턴스별 스냅샷):** CPU 평균/최대(%) · 연결수 평균/최대 · 여유 저장공간 · 여유 메모리 · swap · IOPS · DB Load 등.
- **시계열:** CPU · 여유메모리 · DB Load · 연결수(상세 조회용).
- **상태 의미:** CPU 고부하·여유공간 부족·연결수 포화 임박=위험/주의.

## 3. 운영 인사이트 (룰 + AI)

서로 다른 도메인의 스냅샷을 가로질러 **"지금 주목할 신호(finding)"** 를 자동 추출하는 기능.

- **룰 엔진(결정적):** 임계/상태/추세 규칙으로 신호를 만든다.
  각 신호 = **심각도**(`critical`/`warning`/`info`) · **영역**(알람·EC2·RDS·CDN) · **제목** · **근거 수치**(예: `평균 87.0% ≥ 80%`) · 발생시각.
  대표 임계: CPU 지속 80%·피크 90%, CPU 크레딧 <20, RDS 여유공간 <5GB(위험)/<15GB(주의), RDS 연결수 ≥80, CDN 5xx >1%(위험)·4xx >5%(주의) 등.
- **AI 종합 코멘트(Gemini):** 위 신호 목록**만** 근거로 "무엇부터 봐야 하는지(우선순위·가능한 원인·권장 조치)"를 3~5문장 한국어로 요약. **목록에 없는 수치·원인은 만들지 않는다(환각 금지).**
- **요약 카운트:** critical/warning/info 개수 + 총계.
- **신호가 없을 때:** "주목할 이상 신호 없음(모든 지표 정상)" 을 분명히 보여줘야 한다.

> 시각화 관점: 이건 **"심각도 정렬된 신호 리스트 + 한 줄 종합 진단"** 이라는 데이터 성격을 가진다. 표현 방식은 디자이너 자유.

## 4. 운영봇 채팅 (자연어 Q&A)

- **무엇:** 운영자가 한국어로 물으면, **현재 수집된 캐시 데이터를 근거로** Gemini가 한국어로 답한다.
- **답변 근거(컨텍스트):** 알람 요약 · 최근 24h 가동률 · 최근 7일 트래픽(총요청·사용자·상태분포·상위EP) · DB 상태. **데이터에 없는 건 "모른다"고 답하도록** 설계됨.
- **입출력:** 질문 텍스트(최대 1000자) 1건 → 답변 텍스트 1건(또는 오류 메시지). 멀티턴 메모리는 없음(매 질문 독립).
- **예시 질문(운영자가 자주 묻는 것):** "지금 알람 상태 어때?", "이번주 많이 쓴 API는?", "DB 상태 어때?"
- **모델:** Google Gemini(2.5 Flash). 키 미설정/호출 실패 시 오류 메시지로 graceful 처리.

> 채팅의 **UI/배치/대화 흐름 표현은 전적으로 디자이너 위임.** 위는 "무엇을 묻고 무엇이 돌아오는가"만 규정한다.

## 5. 화면 공통 동작 (상태·컨트롤)

디자인이 반드시 **수용**해야 하는 기능적 상태/컨트롤(형태는 자유):

- **데이터 상태 4종:** 로딩 / 비어있음(수집 전) / 오류 / 정상 — 모든 패널이 이 네 상태를 가진다.
- **수동 새로고침 + "마지막 갱신" 시각** 노출.
- **오래된 데이터 경고(stale):** 수집이 주기×2 넘게 밀리면 "데이터가 오래됨"을 알려야 한다.
- **기간 전환 1/7/30일:** 트래픽·응답품질·가동률에 적용.
- **상세 들여다보기:** 호스트/DB 인스턴스 등 개별 항목의 시계열 상세를 열어볼 수 있어야 한다(드릴다운 — 표현 방식 자유).
- **숫자 가독성:** 큰 수치는 자리수 정렬(tabular) 권장.
- **시각:** 전부 KST 표기.

## 6. 시각화로 풀어야 할 "데이터 성격" 요약

디자이너가 표현 방식을 고를 수 있게, 각 데이터의 **성격**만 정리(차트 지정 아님):

| 데이터 | 성격 | 핵심 결정 |
|---|---|---|
| 알람 | 상태 + 심각도 + 조건 | 지금 위험한가? 무슨 조건? |
| 알람 이력 | 상태 전이 시퀀스(시간) | 언제부터/얼마나 자주 울렸나? |
| 가동률 | 비율(%) 시계열, 평선에 가까움 | 떨어진 구간이 있나? |
| 응답시간 | ms 시계열(avg/p95) | 느려진 구간이 있나? |
| 트래픽 총량/사용자 | 단일 큰 수치(기간) | 규모 감 |
| 상위 엔드포인트 | 랭킹(이름+수치) | 무엇이 가장 많이 쓰이나? |
| 시간대별 요청 | 시계열/분포 | 언제 몰리나? |
| 상태코드 분포 | 비율/구성 | 에러 비중은? |
| 에러 위치 | 랭킹(코드+URL+건수) | 어디가 깨지나? |
| 호스트/DB/CDN 지표 | 인스턴스별 현재값(여러 지표) + 시계열 | 자원이 버티나? 추세는? |
| 인사이트 | 심각도 정렬 신호 리스트 + 종합 진단 | 무엇부터? |

## 7. 디자이너에게 위임하는 범위 (명시)

아래는 **전부 백지에서 새로 설계** — 현재 구현을 참고하지 말 것:
- 정보구조(IA)·**메뉴 구성과 배치**
- **화면(페이지) 구성**과 흐름, 도메인 묶음/분리 방식
- 각 데이터의 **시각화 형태**(차트·표·카드·리스트 무엇이든)
- **채팅 UI**의 위치·형태·대화 표현
- 개요/요약 화면을 둘지, 둔다면 무엇을 먼저 보여줄지
- 반응형·밀도(정보 밀집 vs 여백)·인터랙션·모션

지켜야 할 것은 **§2~§5의 기능·데이터·상태**와 `TOSS-DESIGN.md`의 **시각 언어**뿐이다.
